# Day33 - G474 Motor Driver Bring-up and Low-level Migration

## Overview

Today marked the beginning of the second week of the STM32-based AGV chassis project.

The main task was to migrate the low-level motor driver control from the previous STM32F401RE baseline platform to the new STM32G474RE competition platform.

Instead of directly copying the old F401 project, the migration was restarted from the lowest hardware layer. This helped avoid carrying old configuration issues into the new platform.

## Main Work

### 1. G474 basic bring-up

The NUCLEO-G474RE board was first tested with a minimum project.

The basic bring-up included:

- creating a G474 CubeIDE project
- verifying build and flash process
- confirming board LED behavior
- confirming the board could run normally through USB power

This confirmed that the G474 development environment was usable.

### 2. Arduino header and pin mapping check

The G474 Arduino connector mapping was checked carefully before wiring.

The final motor driver pin plan was:

| Function | G474 Arduino Pin | MCU Pin | Usage |
|---|---|---|---|
| PWMA | D5 | PB4 | TIM3_CH1 PWM |
| PWMB | D6 | PB10 | TIM2_CH3 PWM |
| AIN1 | A1 | PA1 | GPIO Output |
| AIN2 | A2 | PA4 | GPIO Output |
| BIN1 | A3 | PB0 | GPIO Output |
| BIN2 | D4 | PB5 | GPIO Output |
| STBY | D12 | PA6 | GPIO Output |

The original F401 wiring could not be copied directly because the PWM-capable Arduino pins on G474 were different.

### 3. Power issue debugging

A major part of today's work was debugging the power layer.

At first, the G474 3.3V output seemed abnormal when measured with a multimeter. After removing external wires and rechecking the board power pins, the 3.3V and GND measurement became normal.

The final power structure was clarified as:

- G474 powered by USB during debugging
- TB6612 VCC powered by G474 3.3V
- TB6612 VM powered by battery
- G474 GND, TB6612 GND, and battery negative connected together
- encoder power not connected yet

This confirmed the importance of checking power and ground before debugging motor control.

### 4. TB6612 motor driver wiring

The TB6612 motor driver was wired to the G474 according to the new pin plan.

The final motor driver wiring was:

| G474 | TB6612 |
|---|---|
| D5 | PWMA |
| D6 | PWMB |
| A1 | AIN1 |
| A2 | AIN2 |
| A3 | BIN1 |
| D4 | BIN2 |
| D12 | STBY |
| 3V3 | VCC |
| GND | GND |
| Battery + | VM |
| Battery - | GND |

The test focused only on the motor driver layer. Encoder wiring was intentionally postponed to avoid mixing multiple variables.

### 5. Dual motor open-loop test

A minimum motor test program was written for G474.

The program used:

- TIM3_CH1 on PB4 for PWMA
- TIM2_CH3 on PB10 for PWMB
- GPIO outputs for AIN1, AIN2, BIN1, BIN2, and STBY

The final test result:

- G474 successfully controlled the TB6612 motor driver
- both motors could rotate at low speed
- the basic motor driver layer migration was completed

## Important Problems Found

### 1. Direct migration is risky

The old F401 code included motor control, UART commands, Flash record saving, delayed run logic, and encoder counting.

Directly copying the old project to G474 would have created too many variables at once.

A better method is:

1. G474 basic bring-up
2. motor driver layer
3. encoder layer
4. speed measurement
5. closed-loop control
6. command protocol
7. Flash or upper-computer configuration

### 2. Power and ground are critical

The motor did not simply depend on correct code.

The following conditions had to be verified first:

- G474 3.3V output is normal
- TB6612 VCC receives 3.3V
- TB6612 VM receives battery voltage
- G474, TB6612, and battery share common ground
- STBY is pulled high by G474

This is an important hardware lesson.

### 3. G474 pin mapping must be checked independently

The G474 board should not be treated as a drop-in replacement for F401.

Even though both are Nucleo boards, the usable PWM pins and Arduino header mappings must be checked again.

## Reflection

Today was a real low-level migration day.

The most important gain was not just making the motors rotate on G474, but rebuilding the hardware control chain from the bottom:

power -> ground -> pin mapping -> CubeMX configuration -> PWM output -> TB6612 -> motor motion.

This made the system cleaner than simply copying the old F401 code.

The G474 platform is now ready for the next layer: encoder migration and speed measurement.
