# Day28

## Summary
Today was not a heavy technical development day.  
Instead, I used the time to step back and reassess the project from a team and system perspective.

A new member joined the group, so the team has now expanded from two people to three.  
At this stage, the key issue is no longer just whether the code can run on the current F401RE platform, but how to organize the team, prepare for the transition to the STM32G474RE competition platform, and build a clearer execution structure.

## What was clarified today
- Confirmed that the current development and verification work has mainly been based on the **STM32F401RE**
- Confirmed that the group has **not yet formally divided responsibilities**
- Confirmed that the team plans to **purchase the STM32G474RE tonight**
- Recognized that, as the group leader, I now need to develop not only technical ability, but also **basic leadership and project organization ability**

## Main reflection
The project is entering a new phase.

Previously, the focus was mainly on:
- single-person technical exploration
- bottom-level debugging
- motor control, encoder counting, odometry recording, and experiment workflow improvement

Now, another layer has appeared:
- team coordination
- responsibility assignment
- hardware migration planning
- keeping project progress from becoming chaotic

This means I can no longer think only as the person who writes and tests code.  
I also need to think as the person who keeps the whole project moving in a structured way.

## Current judgment
The migration from **F401RE to G474RE** is not trivial, but it is manageable.

The main challenge is likely not the core logic itself, but:
- board-level differences
- pin mapping
- UART mapping
- timer / PWM reconfiguration
- hardware adaptation and migration workflow

So the next stage should not be treated as “copy the code and continue,” but as a controlled board migration process.

## Team direction for the next step
The most important thing now is to establish a basic division of labor in the 3-person team.

A reasonable structure is:
- **I** take the role of group leader and main system owner
- one member takes responsibility for **hardware procurement and board-level migration**
- one member takes responsibility for **testing records, documentation, and presentation materials**

This is important because the project is no longer a solo prototype effort.  
It is becoming a small engineering collaboration, and that requires clearer ownership.

## Why today still matters
Even though today was lighter technically, it was still meaningful.

Today helped clarify:
- the current project stage
- the transition risk from F401RE to G474RE
- the need for team role definition
- the fact that leadership is also a skill that must be trained through practice

This is part of the project too.  
A system does not become better only through code.  
It also becomes better through better structure, better responsibility assignment, and lower organizational friction.

- G474RE purchase: **planned for tonight**

## Next step
- complete the purchase of the STM32G474RE
- assign clear responsibilities within the 3-person group
- continue using the F401RE as the current working baseline before G474RE arrives
- prepare a structured migration plan from F401RE to G474RE
- begin training leadership through concrete project management actions, not just technical work

## Reflection
Today was a reminder that engineering projects do not scale automatically.

When the team grows, the need for structure grows too.  
If responsibilities remain vague, the project will become harder to control.  
So from this point on, part of my growth is no longer only about debugging embedded code, but also about learning how to lead, coordinate, and reduce confusion inside the team.
