# Day32 - System Understanding and Project Direction Refinement

## Overview

Today was mainly a system-level reflection and research day.  
No new firmware or hardware output was produced, but several important decisions were clarified for the AGV project direction.

## Main Understanding

The project should not rely on stacking many external modules to look advanced.  
Instead, the core value should come from making the AGV chassis closed-loop control as solid, measurable, reproducible, and explainable as possible.

The main direction was refined as:

- algorithm depth
- engineering robustness
- motion data visualization
- platform calibration
- stable demonstration

## Project Positioning

The project should be positioned as a small AGV chassis motion control system based on STM32G474.

The focus is not simply making the car move, but making it:

- controllable
- observable
- repeatable
- explainable
- migratable

A better project direction is:

**STM32G474-based AGV chassis closed-loop motion control, calibration, and robust execution system.**

## Key Technical Direction

The most important technical foundation is encoder-based feedback.

Before adding Bluetooth, upper computer software, SLAM, OLED, ESP32, or more sensors, the encoder speed measurement must first become stable.

The next technical priority is:

1. verify left and right encoder counts
2. confirm direction correctness
3. calculate stable delta count
4. estimate wheel speed
5. prepare for speed closed-loop control

## Innovation Direction

The project innovation should be engineering-oriented instead of being based on unrealistic feature stacking.

Potential innovation points include:

1. lightweight task-based motion command protocol
2. encoder-based dual-wheel speed closed-loop control
3. platform migration and motion parameter calibration mechanism
4. distance and turning error compensation based on real measurements
5. Bluetooth offline task input and status feedback

## Community Research

Domestic project communities were also discussed.

The main research sources are:

- OSHWHub / 立创开源广场: hardware design, PCB, BOM, interface design
- Gitee: domestic STM32 project code and student competition projects
- CSDN / 电子发烧友 / forums: debugging issues and practical solutions
- GitHub: higher-quality project structure and international reference projects

The conclusion is that engineering learning is different from AI or pure research learning.  
For embedded projects, useful information is often hidden in datasheets, open-source hardware, code repositories, debugging records, application notes, and real test data.

## Reflection

Today helped clarify that the project should not blindly follow projects with more modules or more attractive hardware boards.

Other projects with SLAM, OLED, ESP32, or complex PCBs are useful references, but the current project should first build a reliable lower-level AGV chassis controller.

The real competitive point should be:

- stable encoder feedback
- reliable closed-loop control
- clear motion task execution
- measurable error output
- repeatable tests
- good documentation

## Next Step

The next practical step is to return to the technical foundation:

**stabilize encoder speed measurement.**

The immediate goal is to make sure that left and right encoder counts are independent, directions are correct, and speed estimation is trustworthy before moving on to PID, distance control, turning control, Bluetooth input, or upper computer visualization.
