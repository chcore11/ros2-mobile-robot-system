# Day1 - Submission Repository Setup and Competition Material Organization

## Overview

Today started the first day of the competition submission repository.

Although the STM32-based AGV chassis project had already been developed for many days before this point, the previous work mainly focused on firmware bring-up, motor control, encoder reading, G474 migration, UART command control, and open-loop chassis testing.

The goal today was not to add new chassis functions. Instead, the focus was to create a clean and dedicated GitHub repository for the final competition submission.

This repository is separated from the daily development repository. Its purpose is to organize the final technical report, video script, source package notes, submission checklist, images, and stage logs in a way that is suitable for review and submission.

By the end of the day, the submission repository was able to:

- store competition-oriented documents separately
- keep submission materials away from temporary development files
- provide a clear README for the project
- organize technical documents under `docs/`
- organize final submission files under `submit/`
- reserve a place for firmware source and package notes
- reserve a place for project images and diagrams
- record the submission preparation process from Day1
- complete local Git initialization
- complete the first commit
- push the repository to GitHub successfully

This means the project has moved from pure development work to the first stage of competition material packaging.

## Main Work

### 1. Repository purpose definition

A dedicated repository was created for the competition submission.

The repository name is:

```text

```

The Chinese project title is:

```text
基于STM32G4的AGV底盘闭环控制系统
```

The repository is not intended to replace the daily development repository.  
Its main role is to become the official material collection point for the competition.

The repository should mainly contain:

- technical report drafts
- submission checklist
- video script
- source package notes
- firmware package description
- hardware images
- system diagrams
- stage logs

The repository should not contain:

- random temporary test files
- unorganized debug screenshots
- unrelated project files

This distinction is important because a submission repository should be readable by judges, teachers, teammates, and future reviewers.

## Important Problems Found

### 1. Submission repository must not become a trash folder

A dedicated submission repository is useful only if the content is controlled.

If all raw development files, temporary screenshots, test outputs, and unfinished materials are thrown into it, the repository will lose its value.

The submission repository must stay clean.

The current rule should be:

```text
Only put organized material into this repository.
```

### 2. Do not overclaim project progress

The project title is about an AGV chassis closed-loop control system.

However, the current development stage still needs more calibration and testing before the final closed-loop performance can be described confidently.

The documentation should avoid exaggerated statements.

Allowed:

- describe implemented modules
- describe current testing workflow
- describe planned closed-loop control
- describe known issues
- describe calibration needs

Not allowed:

- claim high precision without test data
- claim complete industrial AGV capability
- claim autonomous navigation if it is not implemented
- claim stable closed-loop performance before repeated testing

This is especially important for the technical report.

### 3. Submission material should follow the competition page

The competition page requires several deliverables, including:

- demonstration video
- technical document
- source code package
- authorization file
- repository link if needed
- board photos
- project keywords

Therefore, the repository structure should follow the final upload requirements instead of personal preference.

This is why the `submit/` directory was created early.

## Reflection

Today was not a firmware development day. It was a project packaging day.

This kind of work may look less exciting than making the car move, but it is necessary for a competition project.

A working chassis without clear documentation is hard to evaluate.  
A project with scattered files is hard to submit.  
A system without clean explanation is hard to trust.

The most important gain today was that the project now has a formal submission base.

From this point on, the competition materials can be built step by step instead of being rushed at the end.

Another important lesson is that development and submission are different workflows.

Development work allows temporary files, messy experiments, and frequent debugging.  
Submission work requires clarity, structure, and controlled claims.

The next stage should not be to blindly add more files.  
The next stage should be to fill the most important documents first.
