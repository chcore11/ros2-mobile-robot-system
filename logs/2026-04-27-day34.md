# Day34 - G474 Encoder Integration and Open-loop Chassis Control v1

## Overview

Today continued the second-week work of the STM32-based AGV chassis project.

After completing the G474 motor driver bring-up on Day33, the main task today was to migrate the encoder reading layer and integrate motor control, encoder feedback, and UART command control into a first usable open-loop chassis control baseline.

The goal was not to build closed-loop control immediately. Instead, the focus was to make the G474 platform capable of running a controlled open-loop test and recording reliable encoder data.

By the end of the day, the system was able to:

- read left and right wheel encoder counts
- output encoder total, delta, and count/s through UART
- control both motors through TB6612
- receive UART commands
- run forward for a specified number of seconds
- record the encoder result of the last run
- clear encoder counts and previous records

This means the project has moved from simple motor rotation to a first open-loop chassis control baseline.

## Main Work

### 1. Encoder wiring and pin mapping

The encoder layer was connected to the G474 platform.

The current encoder pin mapping is:

| Function | G474 Arduino Pin | MCU Pin | Usage |
|---|---|---|---|
| Left Encoder A | D11 | PA7 | EXTI interrupt input |
| Left Encoder B | D10 | PB6 | Direction input |
| Right Encoder A | A4 | PC1 | EXTI interrupt input |
| Right Encoder B | A5 | PC0 | Direction input |

The encoder A phase was configured as an external interrupt input.  
The encoder B phase was configured as a normal GPIO input for direction judgment.

The first goal was to verify encoder reading manually before running the motors again. This helped avoid mixing motor control problems with encoder input problems.

### 2. Encoder manual reading test

A minimum encoder reading program was written and tested.

The program printed the following values every 200 ms:

- L_total
- R_total
- L_delta
- R_delta
- L_cps
- R_cps

During testing, the UART output worked correctly, and manual wheel rotation produced valid encoder changes.

The manual test confirmed that:

- rotating the left wheel changed the left encoder count
- rotating the right wheel changed the right encoder count
- delta values returned to zero after the wheel stopped
- both encoder channels could produce count/s data

This confirmed that the G474 encoder input layer was basically working.

### 3. Integer cps output

One issue found during the encoder test was that floating-point printf did not display correctly on STM32 by default.

The original output used float format such as:

```c
%.1f

However, the printed cps value was empty in the serial monitor.

To keep the firmware simple and stable, floating-point printf was not enabled through linker settings. Instead, cps was changed to integer calculation.

Because the sampling period was 200 ms:

cps = delta / 0.2

This was simplified to:

cps = delta * 5

After this change, L_cps and R_cps were displayed correctly.

4. Motor control and encoder reading integration

After the encoder manual reading test passed, the motor control layer from Day33 was combined with the encoder reading layer.

The current motor pin mapping is:

Function	G474 Arduino Pin	MCU Pin	Usage
PWMA	D5	PB4	TIM3_CH1 PWM
PWMB	D6	PB10	TIM2_CH3 PWM
AIN1	A1	PA1	GPIO Output
AIN2	A2	PA4	GPIO Output
BIN1	A3	PB0	GPIO Output
BIN2	D4	PB5	GPIO Output
STBY	D12	PA6	GPIO Output

The integrated firmware now includes:

TB6612 motor control
encoder EXTI counting
UART command input
timed open-loop forward running
run result recording
live encoder data output

This created the first G474 open-loop chassis control program.

5. UART command protocol

A simple UART command protocol was implemented.

Current commands:

Command	Function
1~9	wait 3 seconds, then run forward for N seconds
r	report the last run record
c	clear encoder counts and the last record
s	stop the motors immediately
h	print help information

This command protocol allows the chassis to be tested repeatedly without changing and reflashing the code every time.

The basic test workflow is now:

input command -> wait before run -> motor forward -> encoder count -> motor stop -> print record

This is an important step because it turns the chassis into a repeatable test platform instead of a one-time motor test.

6. Open-loop timed run test

The integrated program was tested through the serial monitor.

A command such as 2 could trigger the car to:

wait for 3 seconds
run forward for 2 seconds
stop automatically
calculate left and right encoder delta
calculate left and right count/s
print the last run record

The test confirmed that:

UART command input worked
both motors could be controlled
encoder counts changed during motor motion
the program could record and report the run result
the r command could print the last record
the c command could clear the encoder counts and record

This means the open-loop testing chain is working on the G474 platform.

Important Problems Found
1. Encoder direction must be unified

During the open-loop run test, one wheel produced positive encoder delta while the other wheel produced negative encoder delta during forward motion.

This means the physical wiring or encoder phase relationship is different between the two wheels.

The correct solution for now is not to immediately rewire the encoder. A better engineering method is to define direction correction coefficients in software:

#define LEFT_ENC_DIR   1
#define RIGHT_ENC_DIR -1

The goal is to make the convention clear:

Forward motion: L_delta > 0 and R_delta > 0
Backward motion: L_delta < 0 and R_delta < 0

This direction convention must be fixed before PWM-speed calibration and closed-loop control.

2. The system is still open-loop

Although the car can now run and record encoder data, it is still not a closed-loop control system.

The current firmware does not adjust PWM based on speed error. It only runs the motors with fixed PWM values and records the encoder result.

This is acceptable for the current stage because PWM-speed calibration must be done before PID control.

3. PWM-speed relationship is still unknown

The system can now output L_cps and R_cps, but the relationship between PWM and wheel speed has not been measured yet.

The next stage should not be PID tuning. The next stage should be open-loop calibration.

The required data includes:

PWM = 300
PWM = 350
PWM = 400
PWM = 450
PWM = 500

For each PWM value, the left and right cps values should be recorded and compared.

4. Reproducibility is now becoming important

The physical car has been mainly developed on one computer so far.

Since the system is about to be handed over to other team members, the firmware must be packaged as a reproducible STM32CubeIDE project.

The handoff should include:

Reflection

Today was a real integration day.

The most important gain was not simply that the car could move. The important progress was that the G474 platform can now complete a full open-loop test cycle:

UART command -> motor control -> encoder feedback -> data record -> report output

This makes the chassis much more useful as an engineering platform.

Compared with simple motor rotation, this version gives observable data. With encoder delta and cps output, the next stage can be based on measured results instead of guessing.

Another important lesson is that migration becomes faster after a reliable baseline exists. The previous F401 work exposed many problems early, such as pin mapping, timer configuration, power wiring, and encoder direction. Because of that, the G474 migration was much cleaner and faster.

However, this version should not be overestimated. It is still an open-loop baseline, not a closed-loop chassis. The next real challenge is calibration.
