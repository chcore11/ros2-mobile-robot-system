# Day21 Log - Encoder Debug and System State Assessment

## Summary
Today I focused on verifying encoder feedback under static conditions.

The system has reached a stage where:
- motor driver can be controlled
- encoder signals can trigger interrupts
- count values are observable through UART

However, the system is currently **not in a reliable state**.

## Observations

### 1. Encoder response exists
- encoder count changes when the wheel moves
- interrupt chain is working

### 2. Unexpected coupling between two wheels
- manually rotating one wheel causes both A and B counts to change
- this indicates potential signal interference, incorrect wiring, or mapping inconsistency

### 3. Manual rotation inconsistency
- initial issue: no response when rotating by hand (fixed)
- current issue: count behavior is unstable / non-isolated

## Key Issue Identified
The system cannot yet guarantee:
- independent encoder measurement per wheel
- reliable direction determination
- stable and monotonic count behavior

This means the system is still in a **feedback wiring / signal integrity debugging stage**, not ready for control logic.

## Engineering Reflection
Today reinforced an important principle:

- “Signal exists” ≠ “Signal is correct”
- “Count changes” ≠ “Feedback is usable”

The system must satisfy:
- isolation
- consistency
- predictability

before moving into closed-loop control.

## Next Step (Planned)
1. isolate encoder channels (test A and B independently)
2. verify pin-to-signal mapping without ambiguity
3. eliminate cross-interference between channels
4. ensure monotonic count behavior under manual rotation

Only after that:
→ move to motor-driven validation  
→ then velocity estimation  
→ then closed-loop control
