# Day10 Log - STM32 Motor Debugging and Power Issue Identification

## Summary
Today’s work focused on restarting the STM32 motor control debugging process from the beginning and trying to bring up the single-motor control chain in a cleaner way.

Instead of continuing with scattered trial-and-error, I re-entered the configuration flow from STM32CubeMX, checked the timer and GPIO setup again, regenerated the project, rewrote the main motor test logic, and aligned the code with the current pin mapping.

Although the motor still did not successfully rotate in the final test, today’s main progress was important:
the problem was narrowed down much more clearly, and the current highest-probability issue is now the motor-side power supply rather than the basic CubeMX or control logic setup.

---

## Work Done

### 1. Rebuilt the configuration flow from CubeMX
Today I restarted the setup process from STM32CubeMX instead of continuing from a confusing intermediate state.

The goal was simplified to the smallest possible test target:

- STM32 outputs one PWM signal
- STM32 outputs two direction control signals
- STM32 outputs one standby enable signal
- TB6612 drives a single motor first

This helped reduce unnecessary variables and made the debugging target much clearer.

---

### 2. Confirmed current pin mapping
Based on the current CubeMX configuration, the control mapping was finalized as:

- PA0 -> TIM2_CH1 -> PWMA
- PA1 -> AIN1
- PA4 -> AIN2
- PA6 -> STBY

This was important because the actual generated code no longer matched the earlier temporary assumption that STBY was on PA5.
After checking the generated GPIO initialization and current project state, PA6 was confirmed as the actual standby control pin.

---

### 3. Verified TIM2 PWM configuration
Today I rechecked the timer configuration in CubeMX and confirmed that TIM2 was configured correctly for PWM output:

- TIM2 Channel 1 = PWM Generation CH1
- Prescaler = 83
- Period = 999
- Initial Pulse = 0

This means the PWM configuration itself is now basically in a usable state for motor testing.

---

### 4. Verified GPIO output configuration
I also checked the GPIO default output level and mode configuration.

The key GPIO pins used for TB6612 control were confirmed as:

- Output Push-Pull
- No pull-up / no pull-down
- Low speed
- Default output level = Low

This makes the startup state safer and avoids unstable behavior immediately after power-on.

---

### 5. Rewrote the motor test logic in main.c
After generating the code, I rewrote the motor test logic to make it match the current pin mapping and debugging goal.

The updated logic included:

- starting TIM2 PWM output
- enabling TB6612 standby
- commanding forward rotation
- stopping
- commanding reverse rotation
- stopping again

This version is much more suitable for motor debugging than a loop that continuously writes the same state without delay.

---

### 6. Identified the current main suspected issue
After checking the software side more carefully, the current conclusion is:

the highest-probability problem is now the motor-side power supply.

Reasons:

- CubeMX configuration is basically reasonable
- GPIO and PWM logic are already much clearer
- main.c test logic has been corrected to match the current hardware mapping
- the motor still cannot rotate under the current setup
- the present power source is still not ideal for driving the motor

At this stage, continuing to repeatedly modify software without changing the motor power condition would likely have low debugging value.

---

### 7. Decided to introduce external motor power
Today I further clarified that the next meaningful step is to test again under an external motor power supply.

The temporary solution chosen is:

- use a 4xAA battery box as a simple external supply for TB6612 VM
- keep STM32 logic power separate
- ensure common ground between STM32 and TB6612

This is a much lower-complexity and lower-risk choice than immediately jumping into a more complicated multi-rail power architecture.

---

## Key Progress

Today’s real progress was not “motor already runs,” but rather:

- the CubeMX setup was rebuilt more cleanly
- the current pin mapping became explicit
- the PWM configuration was revalidated
- the main control logic was corrected
- the debugging direction was narrowed down
- the power supply issue became the primary suspect
- the next step was clarified as external power verification

This is meaningful progress because the system is now much less ambiguous than before.

## Reflection
Today was a typical engineering debugging day.

Even though the final visible result was still “the motor does not rotate yet,” the work was not wasted.
The most valuable thing done today was reducing uncertainty.

Previously, there were still multiple possible causes mixed together:

- CubeMX setup confusion
- GPIO uncertainty
- PWM uncertainty
- code logic mismatch
- power supply weakness

After today, the software-side structure is much clearer, and the current bottleneck is much more likely to be on the hardware power side.

In other words, today’s achievement was:
not solving everything,
but turning a vague problem into a much more concrete one.
