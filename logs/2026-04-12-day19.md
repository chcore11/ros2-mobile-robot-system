# Day19 Log - Full Offline Power Bring-Up and Standalone Mobile Base Breakthrough

## Summary
Today marked a major engineering-level breakthrough in the project.

The most important achievement was successfully enabling the STM32-based mobile robot system to run **fully offline**, without relying on USB power from the computer. After reorganizing the hardware architecture from scratch, the system was able to execute motion logic directly from battery power through the buck converter.

This means the project has now crossed an important boundary:
it is no longer just a development-board experiment connected to a PC, but has started to become a real standalone embedded mobile robot platform.

## Work Done

### 1. Rebuilt the Power Architecture from Scratch
To avoid confusion caused by previously mixed wiring states, I disconnected the system and rebuilt it layer by layer.

The rebuilt offline power chain was:

- 2-cell 18650 battery pack
- LM2596 buck converter
- STM32 NUCLEO-F401RE external power input
- TB6612 motor driver logic and power separation

This process clarified the distinction between:
- battery-side raw voltage
- buck-converted 5V logic supply
- STM32 external power path
- TB6612 logic supply (`VCC`) vs motor supply (`VM`)

### 2. Verified Battery-to-STM32 External Power Supply
A key milestone today was confirming that:

- the battery pack could power the LM2596
- the LM2596 could provide a stable 5V output
- the STM32 board could run from this external 5V supply without USB power

After debugging the board-level power selection issue, I found that the jumper settings had to be changed for external supply mode. Once corrected, the board was finally able to run directly from the battery-powered buck converter.

This was the core breakthrough of the day.

### 3. Understood the Role of Board Jumpers in External Supply
A major lesson today was learning that external power on the NUCLEO board is not only about wiring, but also about board-level jumper configuration.

I had initially overlooked the power-selection jumpers, which prevented the board from entering the expected external supply path. After correcting the jumper configuration, the offline power setup finally worked.

This gave me a much clearer understanding of:
- board-level power routing
- the difference between software configuration and hardware power path configuration
- why external power systems must be validated step by step

### 4. Rebuilt the Control Wiring for Full Offline Operation
After the offline STM32 power supply was confirmed, I reconnected the control layer between STM32 and TB6612, including:

- PWM outputs
- direction control pins
- standby control
- logic power line
- common ground structure

This rebuild also reinforced the hardware mapping between MCU pins and board headers.

### 5. Achieved Standalone Motion Capability in Principle
By the end of today, the robot system was able to leave the “computer-dependent” stage.

From an engineering perspective, this means the mobile base now has the fundamental ability to:
- boot independently
- execute pre-programmed motion logic
- move without USB tethering

This is not yet a fully mature mobile robot system, but it is already a true standalone embedded motion platform prototype.

## Key Engineering Gains

### Hardware/System Understanding
Today greatly improved my understanding of:

- layered power architecture
- external board power routing
- the function of jumpers in real hardware systems
- the importance of rebuilding from a clean state when wiring becomes too messy
- separating logic power from motor power
- common-ground requirements in embedded motor systems

### Practical Engineering Thinking
I also reinforced an important engineering principle:

When the system becomes too chaotic, it is often better to disconnect everything and rebuild from the lowest stable layer upward, instead of trying to patch a broken intermediate state.

That approach was critical today.

## Current Status

- TB6612 control wiring rebuilt
- mobile base now capable of standalone execution in principle

Not yet completed:

## What I Learned

- A successful external power system depends on both wiring and jumper configuration
- The NUCLEO board does not automatically accept external power correctly without the proper hardware path selection
- Offline capability is a major structural milestone, not just a minor wiring improvement
- Rebuilding from scratch can be more efficient than trying to repair a confused wiring state
- Real embedded robotics work is as much about power architecture and hardware logic as it is about code
