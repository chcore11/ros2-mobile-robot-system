# Day14 Log - Encoder Polling Optimization Verification

## Summary
Today I optimized the dual-wheel encoder polling test and verified that the previous unstable count behavior at higher manual wheel speed was mainly caused by excessive serial printing and loop delay overhead.

## Work Done

### 1. Reviewed Current Encoder Polling Logic
- Checked the current dual-wheel directional counting code
- Confirmed the existing structure was based on:
  - polling GPIO in `while(1)`
  - counting on A-phase rising edge
  - reading B-phase to judge direction
- Identified two major overhead sources:
  - printing count results on every detected pulse
  - `HAL_Delay(1)` inside the main loop

### 2. Optimized Polling Structure
- Removed per-pulse serial printing from the encoder count path
- Removed `HAL_Delay(1)` from the main loop
- Added periodic serial output using `HAL_GetTick()`
- Changed the print strategy to output counts every 100 ms instead of printing on every count event

### 3. Rebuilt and Verified on Hardware
- Recompiled and flashed the updated firmware to STM32
- Reopened serial monitoring on COM6
- Verified the board could continuously output encoder count values correctly

### 4. Manual Wheel Test
- Tested wheel movement again after optimization
- Compared the new behavior with the previous version
- Observed that encoder count performance was clearly improved
- Fast manual wheel spinning became noticeably more stable than before

## Key Findings
- The previous instability was primarily caused by software overhead in the polling loop rather than a complete failure of the encoder logic
- Frequent UART printing can severely reduce encoder sampling capability in a polling-based design
- Removing loop delay and using periodic printing significantly improves count stability
- The current optimized polling version is usable as a better temporary validation version, but it is still not the final robust architecture
