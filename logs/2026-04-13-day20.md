# Day20 Log - Encoder Wiring Integrated and Feedback Path Brought Into the System

## Summary
Today I extended the offline mobile robot system by integrating the encoder hardware into the current platform.

The main achievement of the day was not a fully working closed-loop system yet, but rather the successful completion of the **encoder-side wiring bring-up** under the existing standalone power architecture.

At this stage, the system has moved beyond pure offline open-loop motion and has started to gain the necessary hardware conditions for motion feedback acquisition.

However, the encoder monitoring / count verification part is still not working correctly yet, so the project should currently be described as:

**offline motion system established + encoder feedback path physically integrated, but not yet fully verified in software.**

## Work Done

### 1. Preserved the Existing Offline Power Architecture
I kept the working standalone power structure from Day19 unchanged instead of introducing new power-side variables.

The current architecture remained:
- battery pack as the main energy source
- LM2596 providing regulated 5V
- STM32 NUCLEO-F401RE powered through external 5V path
- TB6612 logic and motor power paths separated
- system operating without USB power dependency

This prevented the project from regressing into a confused mixed-power state.

### 2. Added Encoder Power Supply Into the System
Today I connected the encoder power lines into the current platform.

Instead of changing the main voltage module to a lower value, I kept the main regulated 5V supply unchanged for the board and used the STM32 3.3V rail to feed the encoders.

This established a cleaner logic-level structure:
- STM32 board powered by regulated 5V
- encoder logic powered from 3.3V
- all grounds kept in a common-ground system

### 3. Completed Encoder Power Distribution by Wire Branching
Because the board only exposes a single 3.3V pin, I used a branching approach to distribute:
- one 3.3V source to both encoder VCC lines
- one GND source to both encoder GND lines

This was an important practical wiring step, because it clarified that one logic power rail can feed multiple low-power modules in parallel.

### 4. Connected Encoder A/B Signal Lines
I then started reconnecting the encoder signal path based on the current pin mapping already used in the code design.

The intended mapping remained:
- Encoder A wheel: PA7 / PB6
- Encoder B wheel: PC1 / PC0

This means the physical feedback channel has now been brought back into the hardware system.

### 5. Clarified the Difference Between “Feedback Wiring Exists” and “Closed Loop Works”
A major understanding gained today was that:

having encoder power and signal wiring connected does **not** mean the system has already become closed-loop.

The actual closed-loop milestone still requires:
- reliable encoder count monitoring
- correct direction/sign behavior
- stable count changes during motion
- software-side use of encoder feedback in control logic

At the moment, the hardware feedback path is present, but the monitoring side is still not functioning correctly, so this is a **feedback integration day**, not a true closed-loop completion day.

## Current Status
Current project status after Day20:
- offline standalone motion system remains functional
- encoder power has been integrated
- encoder A/B signal wiring has been added
- feedback channel has entered the system physically
- monitoring / count verification is still not fully working
- true closed-loop control is not completed yet

## Key Engineering Gains
Today improved my understanding of:
- parallel power distribution from a single 3.3V source
- the relationship between encoder power, ground, and signal wiring
- the difference between hardware integration and software verification
- why “wired” does not automatically mean “working”
- how to incrementally extend a running embedded system without destroying the existing baseline

## What I Learned
- A system is not closed-loop simply because encoders are physically connected
- Feedback only becomes meaningful after count monitoring is verified
- It is better to preserve a working offline baseline and extend it carefully rather than introduce unnecessary new variables
- Hardware bring-up should be evaluated layer by layer: power, signal, observation, then control
- In embedded robotics, the transition from open-loop to closed-loop is not one step, but several verified stages

## Next Step
The next step is very clear:

2. verify that both wheels generate valid count changes
3. verify forward / backward sign correctness
4. confirm stable feedback under low-speed motion
5. only then move into real speed feedback and closed-loop refinement

Day20 was an important integration step, but not the final closed-loop milestone yet.
The feedback hardware path is now in place, and the next battle is software-side verification.
