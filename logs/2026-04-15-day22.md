# Day22 Log - GPIO Configuration Fix and B Motor Output Debug

## Summary
Today I focused on continuing the bring-up of the STM32 + TB6612 + dual encoder system.

The most important progress was identifying a major configuration issue in CubeMX:
the encoder-related GPIO inputs were previously left in floating (`No pull`) state, which caused unstable and misleading feedback behavior.

After correcting the GPIO input configuration, the encoder debugging process became much clearer.

## Work Done

### 1. Fixed GPIO input configuration in CubeMX
I reviewed the STM32CubeMX pin configuration and found that the encoder input pins were not properly configured.

Corrections made:
- encoder EXTI inputs changed to rising-edge trigger
- encoder-related GPIO inputs changed from `No pull` to `Pull-down`

This significantly improved the signal behavior and reduced false triggering.

### 2. Re-verified A wheel encoder feedback
After fixing the GPIO configuration:
- B-side false count issue was largely removed
- A wheel count behavior became much more stable
- the debugging process moved from “chaotic feedback” toward “single-wheel usable feedback”

This confirmed that CubeMX GPIO details were a real root cause in earlier debugging.

### 3. Continued B motor debugging
I then moved back to motor-side debugging and rechecked:
- PWMB / BIN1 / BIN2 mapping
- TIM1 PWM configuration
- AO / BO output path assumptions
- motor-side wiring assumptions

TIM1 configuration in CubeMX was checked again and appeared correct:
- TIM1 CH1 enabled as PWM output
- PA8 assigned correctly
- PB0 / PB5 configured as GPIO outputs

This suggests the software-side PWM path is likely valid.

### 4. Isolated the remaining uncertainty
At this stage, the main unresolved issue is:
- B motor still fails to rotate reliably

Current suspicion is no longer centered on high-level code logic.
The issue is more likely in:
- motor power-side connection
- output channel path
- wire/contact reliability
- physical connection quality of the motor-side lines

## Key Engineering Reflection
Today reinforced an important engineering lesson:

GPIO pin assignment alone is not enough.
Detailed electrical configuration matters:
- pull-up / pull-down
- trigger mode
- signal stability
- floating input prevention

A system may appear “logically correct” while still failing due to low-level configuration details.

Another important lesson:
when software-side configuration has been repeatedly verified,
the next step should be to aggressively question the physical connection layer.

## Current State
What is now clearer:
- encoder GPIO configuration issue was real and has been improved
- A wheel feedback path is much closer to usable
- B motor issue is now narrowed down much closer to the physical output/power path

What remains unresolved:
- B motor still does not rotate reliably

## Next Step
Next session should focus on:
1. rebuilding the B motor output path from scratch
2. rechecking motor-side power wiring and contact quality
3. testing with the smallest possible single-channel validation
4. avoiding further mixed debugging across multiple subsystems at once

The priority is no longer broad exploration, but precise physical-layer verification.
