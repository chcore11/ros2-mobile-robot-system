# Day23 Log - Dual Motor Drive Recovered and Encoder Monitor Restored

## Summary
Today I finally recovered the full basic hardware loop of the STM32 + TB6612 + dual encoder system.

The most important breakthrough was identifying that the core blocking issue was not in the high-level motor logic, but in the CubeMX-side configuration details. After rebuilding the B motor path from scratch and aligning the PWM configuration, both motors were successfully driven again. Then, based on that stable execution layer, I restored encoder monitoring and confirmed that UART could continuously print both A/B counts.

This marks the point where the system moved from repeated uncertain debugging into a usable dual-motor + dual-feedback baseline.

## Work Done

### 1. Rebuilt B motor from the minimum chain
Instead of continuing mixed debugging, I treated B motor as a completely independent subsystem and rebuilt it from scratch.

Checked and validated:
- PWMB -> D7 (PA8)
- BIN1 -> A3 (PB0)
- BIN2 -> D4 (PB5)
- STBY -> D12 (PA6)
- BO1 / BO2 -> B motor power lines

This helped isolate the problem away from the encoder and other unrelated parts.

### 2. Rechecked CubeMX and aligned PWM parameters
I rechecked the TIM1 configuration for the B motor path and found that although the PWM channel assignment was correct, TIM1 and TIM2 were using inconsistent parameter scales before.

Then I aligned TIM1 with TIM2:
- Prescaler = 83
- Counter Period = 999

This made both A/B motor PWM channels share the same scale and removed one major source of ambiguity during debugging.

### 3. Verified B motor independent rotation
After simplifying the code into a B-motor-only minimum test:
- B motor was successfully driven
- the B-side execution chain was confirmed working

This was the key turning point of the whole debugging session.

### 4. Restored dual motor drive
After B motor was confirmed independently, I restored the dual-motor forward test.

Result:
- A motor rotates
- B motor rotates

This confirmed that the basic dual-channel power/control path is working again.

### 5. Restored UART encoder monitoring
On top of the recovered motor drive layer, I restored:
- UART print
- A/B encoder count variables
- EXTI-based encoder callback logic

Result:
- serial monitor continuously prints A/B count values
- both channels now have observable feedback again

This means the system has returned to a usable bring-up baseline:
dual motor actuation + dual encoder monitoring.

## Key Breakthrough
The real root cause was a CubeMX-side configuration issue, which blocked progress for multiple days.

This debugging process reinforced an important lesson:
in embedded bring-up, pin assignment alone is not enough.
Low-level configuration details such as timer scale consistency, GPIO mode, pull configuration, and peripheral setup can silently break the whole system.

## Current State
The system can now:
- drive both motors
- keep a stable dual-motor forward test
- print A/B encoder counts through UART

Today was a real recovery point for the project.
