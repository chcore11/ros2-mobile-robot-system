# Day24 Log - Encoder Validation and Open-Loop Wheel Speed Alignment

## Summary
Today I focused on validating whether the encoder feedback was truly trustworthy and whether the two-wheel chassis could be pushed closer to consistent motion under the current open-loop control setup.

The work was not about adding new high-level logic, but about pressure-testing the current feedback and execution layers step by step:
- first verifying encoder independence
- then verifying direction correctness
- then converting cumulative counts into windowed increments for basic speed observation
- finally performing open-loop PWM compensation to reduce left-right wheel speed mismatch

By the end of today, the system achieved a much cleaner and more convincing baseline:
- encoder feedback is independent and direction-aware
- windowed count increments can reflect wheel motion meaningfully
- low-speed wheel motion became much more balanced after PWM compensation
- after ground testing, the robot could move in an approximately straight line with only a small angular deviation

## Work Done

### 1. Verified encoder independence
Using the dual-motor stop + encoder monitor version, I checked whether each encoder channel responded only to its own wheel.

Results:
- when the wheels were static, counts did not randomly jump
- when only wheel A was rotated by hand, only A changed
- when only wheel B was rotated by hand, only B changed

This confirmed that the current feedback layer no longer had the earlier cross-coupling / false-trigger problem.

### 2. Verified encoder direction behavior
After confirming independence, I further tested forward/reverse manual rotation on each wheel.

Results:
- A wheel direction could be distinguished
- B wheel direction could also be distinguished

This means the encoder feedback is not only “alive”, but already contains usable directional information.

### 3. Built a basic windowed speed observation method
I changed the UART output from only printing cumulative counts:
- A=...
- B=...

to printing count increments inside a fixed time window:
- dA = now_a - last_a_count
- dB = now_b - last_b_count

This effectively turned the encoder from a simple cumulative counter into a basic speed observer.

Results:
- when static, dA and dB stayed near zero
- when manually rotating only A, dA changed while dB stayed near zero
- when manually rotating only B, dB changed while dA stayed near zero

This was an important step from “count exists” to “count increment has physical meaning”.

### 4. Found left-right speed mismatch under equal PWM
With both motors running slowly and windowed increments being printed, I observed that under the same PWM value:
- A wheel was clearly faster than B wheel

This showed that the current main problem had shifted:
not feedback failure, but execution-layer consistency.

### 5. Performed open-loop PWM compensation
Instead of changing multiple things at once, I only adjusted the PWM assignment for the two wheels.

The tuning process was:
- equal PWM first
- then reduce A side
- then refine gradually

After several rounds, the compensation converged to:
- PWM_A = 210
- PWM_B = 220

Results:
- dA and dB became much closer
- low-speed wheel speed mismatch was significantly reduced
- the system responded predictably to parameter tuning

This established the first useful open-loop wheel alignment baseline.

### 6. Ground test
After achieving much better speed agreement in suspended testing, I placed the robot on the ground for a real motion check.

Result:
- the robot could move in an approximately straight line
- only a small angular deviation remained
- no obvious severe drift was observed

This is more meaningful than suspended testing alone, because it confirms that the current compensation also has value under real contact/friction conditions.

## Key Progress
The most important progress today was not “making the car move”, but turning the system into something much more believable and explainable:

- feedback independence verified
- directionality verified
- speed observation method established
- open-loop left/right compensation tuned to a usable level
- ground motion roughly validated

This means the project has moved further from raw bring-up and closer to controllable chassis behavior.
