Day3 focused on building a complete bridge between PC software and embedded hardware.  
The goal was to move from isolated software development to an integrated system involving Python, serial communication, and Arduino.

This marks the first transition from "pure ROS2 simulation thinking" to "real-world control pipeline".

## Result

- PC → Arduino one-way communication established

## Upgrade to Bidirectional Communication

Day3 then extended the system from simple one-way transmission to a command-response architecture.

### Arduino Side

- Implemented serial buffer reading
- Added command parsing:
  - `forward`
  - `back`
  - `left`
  - `right`
- Added response output:
  - `Serial.println("[ACK] ...")`
- Implemented string accumulation with newline termination

### Python Side

- Replaced loop sending with structured commands
- Added serial reading:
  - `ser.readline()`
- Added logging system:
  - `[SEND]`
  - `[RECV]`

## Result

- Full duplex communication achieved:

PC → Serial → Arduino → Response → PC

## Protocol Layer Design (Milestone)

The system evolved from:

raw string transmission

to:

structured command protocol

### Defined Protocol

- Commands:
  - `forward`
  - `back`
  - `left`
  - `right`
- Response format:
  - `[ACK] COMMAND`
- Unknown command handling:
  - `[ACK] UNKNOWN: xxx`

## Significance

- First implementation of a device communication protocol (`V0.1`)
- Built the foundation for future ROS2 → hardware interface

## System Validation

Performed structured validation of the communication pipeline.

### Cold Start Test

- Restarted the system and verified that communication still worked correctly

### Protocol Matching

- Confirmed that command-response mapping was accurate

### Stability Test

- Ran multiple tests without crash

### Error Case

- Encountered:
  - `"Access is denied"` on `COM3`

### Root Cause

- Arduino Serial Monitor was occupying the serial port

## Insight

- Serial port is a single-resource lock system
- Only one side can occupy the port at a time

## Result

## Hardware Integration Attempt (LED Control)

### Goal

Map logical commands to physical outputs.

### Design

Used 4 LEDs to represent direction commands:

- `D2` → `forward`
- `D3` → `left`
- `D4` → `back`
- `D5` → `right`

### Arduino Implementation

- Added `pinMode()` initialization
- Added `digitalWrite()` control
- Added `resetLEDs()` to ensure only one direction LED was active at a time

### Issue Encountered

- LEDs did not light up as expected

### Root Causes

- Missing GND connection
- Circuit was not forming a closed loop
- Breadboard connection understanding was incomplete

## Insight

Hardware requires:

- a closed circuit loop
- correct GND reference
- proper breadboard wiring
